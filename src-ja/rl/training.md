# 学習ループとハイパーパラメータ

## ハイパーパラメータ（FloorCpmOptimizationAgentより）

| パラメータ | 値 | 目的 |
|-----------|-----|------|
| stateSize | 7 | フロア関連メトリクス |
| actionSize | 7 | フロア調整乗数 |
| hiddenSizes | [64, 32] | 典型的より小さい — シンプルな問題 |
| γ (gamma) | 0.95 | 割引率 — フロアの効果はかなり即時的 |
| learningRate | 0.0005 | 保守的 — フロア変更は影響が大きい |
| εStart | 0.8 | 初期探索率 |
| εEnd | 0.05 | 最小探索 |
| εDecay | 0.995 | 学習ステップごとの乗算減衰 |
| bufferSize | 5,000 | リプレイバッファ容量 |
| minBufferSize | 4 | 4遷移後に学習開始 |
| batchSize | 4 | ミニバッチサイズ |
| targetSyncInterval | 50 | 50ステップごとにターゲットネットワークを同期 |
| maxFloorFraction | 0.80 | 最高観測CPMの80%を超えない |
| bidSpreadThreshold | 1.5 | max/min入札比率 > 1.5xの場合のみ学習 |

## 学習サイクル

15分ごと（シミュレーション日では短縮）：

```
1. アクティベーションゲートをチェック：
   - ウィンドウ内にオークションなし？ → スキップ
   - 入札スプレッドなしかつフロアリジェクションなし？ → スキップ（均質市場）

2. 状態をエンコード（ウィンドウメトリクスから7次元）

3. 前回のアクションの結果から報酬を計算

4. 遷移を格納：(prevState, prevAction, reward, newState)

5. リプレイバッファから4のミニバッチをサンプリング

6. Q-ネットワークを学習：
   - 各遷移に対して：target = reward + γ × Q_target(s', argmax Q_online(s'))
   - MSE損失、2層ネットワークを逆伝播
   - 勾配クリッピング ±5.0

7. 50ステップごと：ターゲットネットワークをQ-ネットワークから同期

8. 次のアクションを選択（ε-greedy）

9. アクションを適用：フロアに選択した乗数を乗算

10. [publisherMinFloor, maxObservedCPM × 0.80]にクランプ

11. DQNスナップショットをSiteEntity状態に永続化

12. UpdateFloorCpmをAuctioneerEntityに送信

13. DData（PacingConfig）経由で新しいフロアを公開
```

## 収束タイムライン

1時間あたり4回の観測（15分間隔）で：

| フェーズ | ε | 観測回数 | 動作 |
|---------|---|---------|------|
| 初期 | 0.8 | 0–50 | ほぼランダムなフロア変更、リプレイバッファを充填 |
| 中期 | 0.5 | 50–150 | 学習が効き始め、フロアが最適に向かって推移 |
| 後期 | 0.1 | 300+ | ほぼ活用、最適フロア周辺の微調整 |

実時間では、εは約2〜3週間で0.8から0.05に減衰します。その後もエージェントは適応し続けます — 市場が変化すれば（新しいキャンペーンの参入、予算の変更）、調整します。

## 永続化

DQNの重み（Q-ネットワーク + ターゲットネットワーク）、イプシロン、ステップカウントは`DQNAgent.Snapshot`としてシリアライズされ、`SiteEntity.State.floorAgentSnapshot`に格納されます。サーバー再起動後もエージェントは中断した場所から継続 — 再学習は不要です。
