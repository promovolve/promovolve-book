# Promovolveにおける強化学習

Promovolveは強化学習を1つの目的に使用します：**パブリッシャー側のフロアCPM最適化**。RLエージェントが各パブリッシャーサイトの最低入札価格を調整し、パブリッシャーの収益を最大化します。

## なぜフロアにRLで、入札にはRLを使わないのか？

初期の設計ではキャンペーン入札最適化のRLを検討していました — キャンペーンごとのエージェントが入札を調整するものです。Promovolveの品質調整型オークションでは正直な入札が最適であるため、これは廃止されました。クリアリング価格は競争とクリエイティブ品質から計算され、生の入札額からではありません。入札シェーディングは逆効果です：入札を下げればスコアが下がりインプレッションを失いますが、コストは節約できません（いずれにせよ品質調整型クリアリング価格を支払っていたでしょう）。

フロアCPMは異なります。フロア設定には支配戦略がありません。最適なフロアは現在の市場に依存します — 何社の広告主が入札しているか、いくら支払う意思があるか、予算レベル、クリエイティブ品質。これは常に変化します。静的なフロアは機会損失（低すぎる）か広告主の離脱（高すぎる）のどちらかです。これがRLに適する理由です：実験を通じて適切なレベルを発見します。

## エージェントの仕組み

各パブリッシャーサイトに専用のフロアCPMエージェントがあります。エージェントはSiteEntity内に存在し、15分間隔の観測サイクルで動作します。

### 観測

15分ごとに、エージェントは前のウィンドウからメトリクスを収集します：

| シグナル | ソース | エージェントに伝える内容 |
|----------|--------|--------------------------|
| フィルレート | AuctioneerEntity | オークションが勝者を出しているか？ |
| 入札者数 | AuctioneerEntity | 市場はどの程度競争的か？ |
| 平均落札CPM | AuctioneerEntity | 勝者はいくら支払っているか？ |
| リジェクション率 | AuctioneerEntity | フロアが何件の入札を排除しているか？ |
| 配信収益 | LearningEventLog | 実際の収益はいくらか？（ペーシング後） |
| 予算枯渇 | AdServer | 予算が早すぎるペースで消耗しているか？ |

エージェントはこれらを7次元の状態ベクトルにエンコードし、[0, 2]の範囲に正規化します。

### 判断

エージェントは7つのアクションから1つを選択します：現在のフロアに0.90、0.95、0.98、1.00、1.02、1.05、または1.10を乗算。小さなステップ — フロアは1回の観測で10%以上変動しません。

新しいフロアは範囲内にクランプされます：
- **下限**：パブリッシャーの最低フロア（ダッシュボードで設定 — 「$X以下で在庫を売らない」）
- **上限**：観測された最高入札額の80%（すべての広告主を排除することを防止）

### 報酬

エージェントの報酬シグナルは、オークション時の推定値ではなく、実際の配信収益です：

```
reward = 正規化収益 × フィルレート
       − 予算枯渇ペナルティ
       − ボラティリティペナルティ
```

なぜオークション収益ではなく配信収益なのか？ペーシングコントローラーがオークションと配信の間でインプレッションをスロットルするからです。高いフロアは素晴らしいオークションクリアリング価格を生み出すかもしれませんが、予算の消耗が早すぎてペーシングが配信の80%をスロットルすれば、パブリッシャーの実際の収益はオークションが約束した額のわずかな割合になります。エージェントには実際のシグナルが必要です。

### いつ起動するか

エージェントは市場が最適化を正当化する場合にのみ実行されます：

- **多様な入札**：最高/最低CPM比率 > 1.5倍、またはフロアによって入札がリジェクトされている場合
- **均質な市場**：すべての入札者が同じCPM → エージェントは休止（最適化するものがない）

これによりリプレイバッファがクリーンに保たれます。エージェントはアクションが意味を持つ市場条件でのみ学習します。

## エージェントが学ぶこと

数百回の観測を通じて、エージェントは現在の市場における収益最大化フロアを発見します：

**フロアが低すぎる** → すべてのキャンペーンが資格を持つが、クリアリング価格が低い。フロアが仕事をしていない。

**フロアが高すぎる** → キャンペーンがリジェクトされ、フィルレートが低下し、残ったキャンペーンに競争がない（単独勝者はフロアを支払う）、予算が急速に消耗。

**スイートスポット** → 健全なクリアリング価格のための十分な競争、下位の入札がフィルタリングされ、予算が1日を通して持続。

$2、$4、$6、$8、$10の入札がある市場では：エージェントは$3-4付近に落ち着く可能性があります。これは$2の入札者（あまり競争に貢献していなかった）をフィルタリングしつつ、4つのキャンペーンが積極的に競争する状態を維持します。

## アルゴリズム

エージェントは**Double DQN**（Deep Q-Network）を使用します：

- **Q-ネットワーク**：7入力 → 64 → 32 → 7出力（アクションごとに1つのQ値）
- **ターゲットネットワーク**：Q-ネットワークのコピー、50学習ステップごとに同期
- **リプレイバッファ**：5,000遷移、4のミニバッチでサンプリング
- **エクスプロレーション**：ε-greedy、0.8から0.05まで減衰

7次元の状態と7つのアクションにDQNは過剰か？おそらく。しかし、DQNインフラは（現在は削除された）キャンペーン入札最適化のためにすでに構築されており、確実に動作します。計算コストは無視できます — 15分ごとに小さなネットワークを1回のフォワードパスするだけです。

## 永続化

DQNの重みはSiteEntityの永続状態にシリアライズされます。サーバーが再起動すると、エージェントは学習済みの重みを復元し、中断した場所から続行します。再学習は不要です。

## パブリッシャーのコントロール

パブリッシャーは2つの境界をコントロールします：

- **最低フロア**（サイト作成時に必須）：「$X以下にしない」
- **広告選択優先度**（ディスカバリー/バランス/レベニュー）：スコアリング指数αをコントロール

RLエージェントはこれらの境界内で動作します。パブリッシャーの最低フロアをオーバーライドしたり、広告優先度を変更したりすることはできません。パブリッシャーがレールを敷き、エージェントがその中で最適化します。
