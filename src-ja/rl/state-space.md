# 状態空間

フロアCPMエージェントは、15分の観測ウィンドウから計算される**7次元の状態ベクトル**を観測します。

| 次元 | 特徴量 | 正規化 | エージェントに伝える内容 |
|------|--------|--------|--------------------------|
| 0 | currentFloor / maxObservedCPM | 上限2.0 | フロアは市場に対して高いか低いか？ |
| 1 | fillRate | 生の0–1 | オークションが勝者を出しているか？ |
| 2 | avgWinningCPM / currentFloor | 上限5.0 | 勝者はフロアよりどれだけ高く入札しているか？ |
| 3 | avgBidderCount / 10 | 上限2.0 | 市場はどの程度競争的か？ |
| 4 | servedRevenue / baseline | 上限3.0 | 実際のペーシング後の収益シグナル |
| 5 | rejectionRate | 生の0–1 | フロアが何件の入札を排除しているか？ |
| 6 | budgetExhaustionRate | 生の0–1 | 予算が早すぎるペースで消耗しているか？ |

## 主な設計判断

**ペーシング後の収益（次元4）：** 収益シグナルは`LearningEventLog`が追跡する実際の配信インプレッションから得られ、オークション時のクリアリング価格からではありません。これにより、ペーシングがインプレッションをスロットルするため配信収益に変換されない高いオークション価格にエージェントが騙されることを防ぎます。

**予算枯渇（次元6）：** 高いフロアは予算をより早く消耗させます（単独勝者はフロア価格を支払う）。この次元は、枯渇した予算から収益がゼロに落ちる前に、エージェントに早期警告を与えます。

## 収益のベースライン

オークションあたりの収益（次元4）は`currentFloor / 1000`で正規化されます — フロア価格での1インプレッションの収益。これによりシグナルが絶対的なフロアレベルに依存しなくなります：
- 正規化収益 = 1.0は「オークションあたりちょうどフロア価格を稼いでいる」
- 正規化収益 > 1.0は「競争がクリアリング価格をフロア以上に押し上げている」
- 正規化収益 < 1.0は「フロア価格でも埋まっていない」
