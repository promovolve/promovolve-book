# 第4章: 読者がやって来る

午前10時15分。シンガポールの旅行者が11月の京都旅行を計画しており、検索エンジンでユキの記事を見つける:「東京都東部の紅葉ハイキング」。ページが読み込まれる。記事がレンダリングされる。そしてサイドバーの広告枠を埋める必要がある。

ユキのJavaScriptスニペットがPromovolveにリクエストを送信する:

```
GET /v1/serve?pub=yuki-site&url=https://yukiblog.jp/autumn-hikes&slot=sidebar-1
```

次に起きることは1ミリ秒もかからない。

## ステップ1: ローカルDData検索

このリクエストを処理するAPIノードは、ローカルのDDataレプリカで`yuki-site|sidebar-1`を検索する。これはJVMのメモリ上のハッシュマップ検索であり、ネットワーク呼び出しもデータベースクエリもない。昨晩のオークションからの候補（5分間のre-auctionサイクルで更新済み）がそこにある:

```
Candidates:
  1. Takeshi's Ryokan — $5.00 CPM, Beta(1, 1)  [new, no impressions yet]
  2. Hiking Gear Co   — $4.00 CPM, Beta(1, 1)  [new, no impressions yet]
```

両方のクリエイティブがユキに承認されている。両方とも予算が残っている。両方とも配信準備完了だ。

## ステップ2: コンテンツ鮮度チェック

システムは確認する: このページはまだ新鮮か？ 分類タイムスタンプは8時間前にクロールされたことを示している。ユキのサイトの鮮度ウィンドウは48時間だ。8時間は十分にその範囲内であり、ページは新鮮だ。続行する。

もし記事が先週のもので再クロールされていなければ、候補は期限切れ（TTL 120分）になっており、レスポンスは`204 No Content`、つまり空の広告枠になる。これは設計通りだ: 古いコンテンツは収益化されない。

## ステップ3: ペーシングゲート

クリエイティブを選択する前に、PI controllerが判断する: そもそも配信すべきか？

午前10時15分だ。タケシの旅館キャンペーンは$20の日次予算がある。システムがユキのサイトについて学習したトラフィック形状（JST午前8時〜11時に最もトラフィックが多く、午後8時にもう一つのピーク）に基づくと、午前10時15分までの理想的な支出は予算の約35%、つまり$7だ。キャンペーンはこれまでに$4を使っている。若干アンダーペーシングだ。

PI controllerがスロットル確率を計算する: 0.12（ペースを維持するためリクエストの12%をスキップ）。乱数が引かれる: 0.47。これは0.12を上回っているので、このリクエストはゲートを通過する。

もしキャンペーンが過剰支出していたら、例えば午前10時15分までに$12使っていたら、スロットルはもっと高く、おそらく0.65になり、より多くのリクエストがスキップされる。キャンペーンは減速し、残りの予算を午後にかけて引き延ばす。

これはクリエイティブ選択の前に行われる。ゲートがリクエストを拒否した場合、Thompson Samplingは実行されない。スロットルされたインプレッションに探索は浪費されない。

## ステップ4: Thompson Sampling

システムがどのクリエイティブを表示するか選択する。両方の候補は全くの新規で、インプレッションゼロ、クリックゼロだ。Beta distributionは両方とも`Beta(1, 1)`、一様分布だ。

Thompson Samplingが各候補からランダムなサンプルを引く:

```
Takeshi's Ryokan: sample from Beta(1, 1) → 0.72
  score = 0.72 × log(1 + 5.00) = 0.72 × 1.79 = 1.29

Hiking Gear Co:   sample from Beta(1, 1) → 0.34
  score = 0.34 × log(1 + 4.00) = 0.34 × 1.61 = 0.55
```

タケシの旅館がこのラウンドで勝つ。より良いからではなく、まだ誰にも分からないが、ランダムサンプルがたまたま高かったからだ。次回はハイキング用品広告が勝つかもしれない。両方とも`Beta(1, 1)`なので、ほぼコイントスに近く、CPMでわずかに重み付けされている。

これは純粋な探索だ。次の100インプレッションで、システムは読者がどのクリエイティブを実際にクリックするかを学習し、ランダム性は情報に基づいた選択に道を譲る。

## ステップ5: 予算の予約

広告を配信する前に、システムは支出を予約する。タケシのキャンペーンのCampaignEntityが`TryReserve`リクエストを受信する:

- 金額: $5.00 / 1000 = $0.005（CPM $5での1インプレッション）
- 予算残高: $16.00
- 結果: 予約完了。予算は$15.995になった。

もし予算が枯渇していたら、レスポンスは`InsufficientBudget`となり、Thompson Samplingは次の候補（Hiking Gear Co）を試す。すべての候補が枯渇していたら、レスポンスは`204 No Content`だ。グレースフルデグラデーション、エラーなし。

予約はCampaignEntityのエフェメラルバッファに記録される。RL agentがインプレッションを記録する: `recordImpression($0.005)`と`recordBidOpportunity(won: true)`。

## ステップ6: レスポンス

APIが1ミリ秒以内にJSONレスポンスを返す:

```json
{
  "assetUrl": "https://cdn.promovolve.dev/ryokan-hakone-300x250.jpg",
  "mime": "image/jpeg",
  "width": 300,
  "height": 250,
  "clickUrl": "https://api.promovolve.dev/v1/click?pub=yuki-site&...",
  "impUrl": "https://api.promovolve.dev/v1/imp?pub=yuki-site&...",
  "creativeId": "ryokan-001",
  "version": 1711090800000
}
```

JavaScriptスニペットが画像をレンダリングし、インプレッションピクセル（`impUrl`）を発火する。トラッキングURLはHMAC署名されており、偽造や改ざんはできない。

## 読者が見るもの

紅葉に囲まれた山の中の旅館の写真が、京都の秋のハイキングに関する記事の横に表示される。広告は関連性がある。コンテキストに合っている。読者はそれを押し付けがましい意味での「広告」とさえ認識しないかもしれない。まるで推薦のように感じる。

これこそ私たちが構築しようとしたものだ: ウェブ上の雑誌体験。

読者は記事を読む。旅館の写真に気づく。「旅行にいいかもしれない」と思う。クリックする。
