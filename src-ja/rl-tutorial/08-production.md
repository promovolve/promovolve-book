# 本番環境での訓練: episode、永続化、日次リセット

ほとんどのRLチュートリアルは、訓練ループとrewardの上昇グラフで終わります。実際のシステムは、その後に来るすべてのことに対処しなければなりません: 深夜に何が起こるか？サーバーが再起動したら何が起こるか？エージェントが何も知らない初日に何が起こるか？この章では、チュートリアルが通常省略する本番環境での懸念事項を扱います。

## episodeは日単位

教科書の定式化では、RL episodeはエージェントがterminal stateに到達するまでのstateとactionの列です。Promovolveでは、各配信日が1つのepisodeです。エージェントは入札倍率1.0で1日を開始し、1日を通して15分ごとに観測を行い、episodeは深夜に（または予算が尽きた場合はその時点で）終了します。

`resetDay()`メソッドがepisode間の遷移を処理します:

```scala
def resetDay(): Unit = {
  // Store terminal transition only if we have both a pending state AND a real action
  // (prevAction is None if observe() was never called this day — no transition to record)
  for {
    ps <- prevState
    pa <- prevAction
  } {
    val terminalState = Array.fill(config.dqnConfig.stateSize)(0.0)
    val terminalReward = windowClicks.toDouble
    dqn.store(ps, pa, terminalReward, terminalState, done = true)
  }

  _bidMultiplier = 1.0
  prevObservation = None
  prevState = None
  prevAction = None
  windowImpressions = 0
  windowClicks = 0
  windowSpend = 0.0
  windowBidOpportunities = 0
  windowWins = 0
  dayImpressions = 0
  dayClicks = 0
  daySpend = 0.0
  dayObservations = 0
  dayRewardSum = 0.0
}
```

ここでは多くのことが起こっているので、分解して見ていきましょう。

**terminal transition。** 1日の最後の観測はactionを選びましたが、通常の`observe()`呼び出し中に結果を見ることはありませんでした。`resetDay()`メソッドはこのギャップを埋めるために最後のtransitionを1つ格納します: 最後のstateとaction、最終ウィンドウで蓄積されたクリックに基づくreward、ゼロ埋めされたterminal state、そして`done = true`です。これがなければ、エージェントは毎日の終わりの情報を失うことになります -- 予算枯渇や過少消化が現れやすい、最も興味深い部分であることが多いのに。

`for`内包表記は、その日に`observe()`が一度も呼ばれなかった場合（キャンペーンが一時停止されていた場合など）に対するガードです。その場合、保留中のstateやactionがないため、記録するものはありません。

**倍率のリセット。** 入札倍率は1.0に戻ります。各日は新たに始まります。市場の状況、トラフィックパターン、競争環境が今日は全く異なる可能性があるため、エージェントは昨日の倍率を引き継ぎません。

**すべてのカウンターのクリア。** ウィンドウカウンター、日次カウンター、前のstate/actionの参照がすべてゼロにされます。

**学習済みの重みは保持。** これが重要な部分です: `resetDay()`は`DQNAgent`には触れません。Q-networkの重み、target networkの重み、replay bufferに格納された経験、現在のepsilon -- これらすべてが日をまたいで生き残ります。エージェントは生のカウンターに関しては今日何が起こったかを忘れますが、入札方法について学んだことはすべて覚えています。

## 永続化: 再起動への耐性

エージェントの学習済みの重みは、プロセスの再起動を生き延びなければなりません。サーバークラッシュやデプロイメントが数週間の学習を消去してはなりません。Promovolveはsnapshotを通じてこれを処理します。

`DQNAgent.snapshot()`は完全な学習可能状態をキャプチャします:

```scala
def snapshot(): DQNAgent.Snapshot =
  DQNAgent.Snapshot(
    config = config,
    qNetworkSnapshot = qNetwork.serialize(),
    targetNetworkSnapshot = targetNetwork.serialize(),
    epsilon = epsilon,
    totalSteps = totalSteps,
    trainSteps = trainSteps
  )
```

各ニューラルネットワークは、すべての重みとバイアスを1次元配列にフラット化してシリアライズします:

```scala
def serialize(): DenseNetwork.Snapshot = {
  val flatWeights = weights.flatMap(_.flatMap(_.toSeq)).toArray
  val flatBiases = biases.flatMap(_.toSeq).toArray
  DenseNetwork.Snapshot(layerSizes, flatWeights, flatBiases)
}
```

8-64-64-7のネットワークの場合、`(8*64) + (64*64) + (64*7) = 512 + 4096 + 448 = 5,056`個の重み値と`64 + 64 + 7 = 135`個のバイアス値です。2つのネットワークで合計約10,382個の浮動小数点数。doubleとして格納すると約83 KB -- 頻繁に永続化しても問題ないサイズです。

これらのsnapshotはPekkoのdurable stateメカニズムを通じてPostgreSQLにバックアップされて格納されます。再起動時に、`DQNAgent.fromSnapshot()`がエージェントを再構築します:

```scala
def fromSnapshot(snapshot: Snapshot, rng: Random): DQNAgent = {
  val agent = new DQNAgent(snapshot.config, rng)
  agent.restore(snapshot)
  agent
}
```

そして`restore()`がすべてを元に戻します:

```scala
def restore(snap: DQNAgent.Snapshot): Unit = {
  qNetwork.restore(snap.qNetworkSnapshot)
  targetNetwork.restore(snap.targetNetworkSnapshot)
  epsilon = snap.epsilon
  totalSteps = snap.totalSteps
  trainSteps = snap.trainSteps
}
```

**replay bufferは永続化されません。** これは意図的な選択です。bufferは最大10,000個の生のtransitionを保持しており、シリアライズするとはるかに大きくなります。そしてさらに重要なことに、それは必要ありません。ニューラルネットワークの重みには、それらすべてのtransitionから蒸留された知識が既に含まれています。再起動後、bufferは空の状態から始まり、新しい経験で自然に補充されます。32個の新しいtransitionが蓄積されると（`minBufferSize`）訓練が再開されますが、15分間隔で約8時間かかります。その間、エージェントは再起動前の重みを使って意思決定を行い続けます -- bufferに十分なサンプルが集まるまで新しいデータからは学びません。

## inference-onlyモード

エージェントが十分に訓練され、さらなる探索なしに学習済みの方策を実行するだけでよい場合があります。`inferenceOnly`フラグがこれを制御します:

```scala
val action =
  if (config.inferenceOnly) dqn.selectGreedy(state)
  else dqn.selectAction(state)
```

`inferenceOnly`がtrueの場合、エージェントは`selectAction()`の代わりに`selectGreedy()`を呼び出します。違いはシンプルです: `selectGreedy()`は常にQ-valueが最も高いactionを選び、`selectAction()`はepsilon-greedyを使用して時折ランダムなactionを選びます。

```scala
def selectGreedy(state: Array[Double]): Int =
  argMax(qNetwork.forward(state))
```

inference-onlyモードでも、エージェントは引き続き`observe()`を呼び出し、transitionを記録し、日次統計を蓄積します。ただ探索をしないだけです。これは、広告主が予測可能で安定した動作を望み、エージェントが既に良い方策を学習しているキャンペーンに有用です。

## cold start問題

新規キャンペーンのエージェントは以下の状態です:

- **ランダムな重み**（Xavier初期化）。生成するQ-valueは本質的にノイズです。どのactionが良いか全く分かりません。
- **epsilon = 1.0。** すべてのactionがランダムに選ばれます。エージェントはQ-valueを参照しません。
- **空のreplay buffer。** エージェントが訓練したくても、学習するtransitionがありません。

悪そうに聞こえますが、実際には問題ありません。何も知らないとき、ランダムな探索は正しい戦略です。エージェントはさまざまな入札調整を試し、結果を観察し、経験を蓄積し始めます。

新規キャンペーンのタイムラインを以下に示します:

**0-8時間（観測1-32）。** エージェントはランダムなactionを取り、transitionを蓄積します。replay bufferが最小サイズ32に達していないため、訓練は行われません。入札倍率は[0.5, 2.0]の範囲内でランダムに変動します。これは無駄ですが避けられません -- エージェントは学習する前にデータが必要です。

**8-24時間（観測33-96）。** 訓練が開始されます。各観測が訓練ステップをトリガーするようになります: bufferから32個のtransitionをサンプリングし、Double DQN目標を計算し、Q-networkを更新します。epsilonはまだ高い（64訓練ステップ後で約0.85）ため、ほとんどのactionはまだランダムですが、Q-valueは意味を持ち始めています。

**2-7日目。** epsilonは最初の1週間で約0.85から約0.05に低下します。エージェントはランダムな探索よりも学習済みのQ-valueにますます依存するようになります。認識可能な行動が現れ始めます: ペースを上回っているときは倍率を下げ、過少消化のときは上げ、勝率が既に高いときは引き下げる傾向があります。

**2週目以降。** epsilonは0.05の下限に達しています。エージェントは95%活用、5%探索です。キャンペーンのトラフィックパターン -- インプレッションが豊富なとき、希少なとき、1日の異なる時間帯で競合他社がどれだけ積極的か -- に合わせた安定した方策を発達させています。倍率はランダムに変動するのではなく、滑らかな日次パターンに従います。予算ペーシングは厳密です。ドルあたりのクリックは、キャンペーンのターゲティングとクリエイティブの品質に対してほぼ最適です。

**継続的に。** エージェントは学習と適応を続けますが、変化は漸進的です。5%の探索により、トラフィックパターンが変化した場合（季節変動、新しい競合他社、クリエイティブローテーションの変更）に気づくことが保証されます。キャンペーン設定の大幅な変更（新しい予算、新しいターゲティング）は、エージェントが部分的に再学習しなければならない新しい環境を事実上作り出します。

重要なポイント: RLは遅いということです。結果を出すまでに数日間のリアルデータが必要です。これは、データセットで数分で訓練できる教師あり学習とは根本的に異なります。エージェントは実際の市場と対話し、実際の結果を観察し、実際の結果から学ばなければなりません。近道はありません。

## 複数のエージェント、1つの市場

Promovolveのすべてのキャンペーンは独自の`BidOptimizationAgent`を持っています。50のアクティブなキャンペーンがあれば、50の独立したエージェントがあり、それぞれが独自のQ-network、replay buffer、epsilonスケジュールを持ちます。互いに通信せず、互いの存在を知りません。

これは興味深い状況を作り出します。任意のエージェントの視点からは、環境は非定常です: 他の49のエージェントも入札を調整しており、オークションのダイナミクス、勝率、実効価格が変化します。昨日うまくいったactionが、競合他社が一晩で入札を上げたために今日はうまくいかないかもしれません。

非定常性にもかかわらず、なぜこれが機能するのでしょうか？いくつかの要因が安定性を維持しています:

**小さな調整。** 各actionは倍率を0.7から1.4の範囲で変更します。単一のエージェントが1ステップで市場を劇的に変えることはできません。

**遅い更新。** エージェントは15分ごとに観測します。市場は次の調整ラウンドの前に変化を吸収する時間があります。

**継続的な学習。** epsilonはゼロにならない（0.05で下限）ため、エージェントは常にある程度の探索を行い、変化する条件に適応できます。競合他社が積極的に入札し始めたら、エージェントは勝率の低下を観測し、調整を学びます。

**安定化装置としてのPI pacing。** PI pacingコントローラーは各15分ウィンドウ内の市場変動への高速対応を処理します。RLはゆっくりとした戦略的調整を処理します。このタイムスケールの分離が振動を防ぎます。

結果として、多くの学習エージェントが共存し、それぞれが自身のキャンペーンに対して良い方策に徐々に収束するシステムが得られます。ゲーム理論的な意味でのNash均衡ではありませんが、実際の広告市場には十分に安定しています。

## エージェントが数週間で実際に学ぶこと

新しいキャンペーンをデプロイし、エージェントの行動を時間の経過とともに観察すると、おおよそ次のようなものが見えます:

**1週目: ほぼランダム。** epsilonはその週のほとんどで約0.7です。エージェントは多くのランダムな入札調整を行います。日ごとのrewardはノイズが多いです。倍率の時系列はクランプ付きのランダムウォークのように見えます。ある日は予算を早期に使い果たし、別の日は過少消化になることがあります。

**2週目: 活用の開始。** epsilonは約0.3に低下しています。エージェントは70%の確率で学習済みの方策を選択しています。パターンが見え始めます: 予算消化がペースを上回っているときは倍率が下がる傾向があり、遅れているときは上がります。予算枯渇ペナルティの発生頻度が減ります。日次rewardは上昇傾向を示し始めます。

**3-4週目: 安定した方策。** epsilonは0.05の下限に達しています。エージェントは95%活用です。キャンペーンのトラフィックパターン -- インプレッションが豊富なとき、希少なとき、1日の異なる時間帯で競合他社がどれだけ積極的か -- を学習しています。倍率はランダムに変動するのではなく、滑らかな日次パターンに従います。予算ペーシングは厳密です。ドルあたりのクリックは、キャンペーンのターゲティングとクリエイティブの品質に対してほぼ最適です。

**継続中。** エージェントは学習と適応を続けますが、変化は漸進的です。5%の探索により、トラフィックパターンが変化した場合（季節変動、新しい競合他社、クリエイティブローテーションの変更）に気づくことが保証されます。キャンペーン設定の大幅な変更（新しい予算、新しいターゲティング）は、エージェントが部分的に再学習しなければならない新しい環境を事実上作り出します。

## モニタリング: 学習曲線を見守る

`dayStats`メソッドはダッシュボード用の日次サマリーを提供します:

```scala
def dayStats: BidOptimizationAgent.DayStats = BidOptimizationAgent.DayStats(
  impressions = dayImpressions,
  clicks = dayClicks,
  spend = daySpend,
  observations = dayObservations,
  totalReward = dayRewardSum
)
```

これらの値を日ごとにプロットすることで、エージェントの学習を観察できます:

- **Total reward**は最初の1-2週間で上昇傾向を示し、その後プラトーに達するはずです。初日からフラットな場合、エージェントが訓練されていない可能性があります（`inferenceOnly`がfalseで`observe()`が呼ばれていることを確認してください）。
- **クリック数**はエージェントがより効果的に入札を学ぶにつれて増加するはずです。
- **CTR**（`clicks / impressions`）は、ユーザーのエンゲージメントが高い時間帯にオークションに勝つことを学んだ場合に増加する可能性があります。
- **クリック単価**（`spend / clicks`）は、エージェントが過払いせずに勝つのに十分な額だけ入札することが上手くなるにつれて、減少または安定するはずです。
- **観測数**は1日あたり約96（24時間 x 1時間あたり4観測）であるべきです。著しく少ない場合、キャンペーンが一時停止されているか予算不足になっていることを意味します。

より深いデバッグのために、`currentQValues`はエージェントが現在のstateについて何を「考えている」かを示します:

```scala
def currentQValues: Option[Array[Double]] = prevState.map(dqn.qValues)
```

これは最新のstateに基づく7つのactionそれぞれのQ-valueを返します。エージェントが十分に訓練されていれば、明確な差別化が見えるはずです: 1つか2つのactionが残りよりも著しく高いQ-valueを持っています。すべてのQ-valueがほぼ同じなら、エージェントはまだaction間の区別を学んでいません -- より多くの訓練時間が必要です。

`epsilon`をチェックして、エージェントが探索スケジュールのどこにいるかも確認できます:

```scala
def epsilon: Double = dqn.currentEpsilon
```

epsilonがまだ高い（0.5以上）場合、エージェントはほぼ探索中であり、一貫した行動はまだ期待できません。epsilonが0.05であれば、エージェントは学習済みの方策を実行しており、不安定な行動は方策自体がより多くの訓練データを必要としていること -- または環境が変化したことの兆候です。

## まとめ

本番環境のRLは研究のRLとは異なる分野です。コアアルゴリズム（Double DQN）は同じですが、その周囲のエンジニアリングが極めて重要です:

- **episodeは日にマッピングされます。** 1日の終わりのterminal transitionが学習ループを閉じます。倍率はリセットされ、重みは永続化されます。
- **永続化は軽量です。** 2つのネットワークをフラット配列にシリアライズし、epsilonとステップカウントを加えます。キャンペーンあたり約83 KB。replay bufferはネットワークの重みが学習内容を既にエンコードしているため永続化されません。
- **cold startは設計によって処理されます。** epsilon = 1.0でのランダム探索は、何も知らないときの正しい戦略です。訓練は約8時間の経験の後に開始されます。
- **マルチエージェントの安定性は、遅く小さな調整から生まれます。** 各エージェントは独立して学習しますが、変更が漸進的でPI pacingコントローラーが高速対応を処理するため、市場は安定を保ちます。
- **モニタリングは不可欠です。** 日次reward、クリック数、CPC、Q-valueの検査により、運用者は学習が進んでいることを確認し、問題を早期に発見できます。

RLエージェントは独立したシステムではありません。PI pacingが短期的な配信平滑化を処理し、Thompson Samplingが配信時のクリエイティブ選択を処理し、RLエージェントが日単位・週単位で全体的な入札戦略を最適化する、より大きなアーキテクチャの1つのコンポーネントです。各ピースは異なるタイムスケールで動作し、合わさることでレスポンシブかつ適応的なシステムを生み出します。
