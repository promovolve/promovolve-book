# なぜ Promovolve なのか？

## 問題：プログラマティック広告はパブリッシャーのために作られていない

現代のプログラマティック広告スタック — SSP、DSP、exchange、DMP — は、大手広告主の課題を解決するために設計された。数百万のウェブサイトを横断して、リアルタイムに適切なユーザーへリーチする方法である。その目的には見事に成功した。しかしその結果、ほとんどのパブリッシャーの利益に反するシステムが生まれてしまった。

### パブリッシャーは自社の在庫に対するコントロールを失った

ユーザーがパブリッシャーのサイトを訪れたとき、広告の意思決定はまったく別の場所で行われる。パブリッシャーの SSP がビッドリクエストを exchange に送り、exchange がそれを数十の DSP にブロードキャストし、DSP がユーザープロファイルを参照して入札アルゴリズムを実行し、レスポンスを返す — すべて100ミリ秒以内に。パブリッシャーは入札を見ることはない。どの広告が掲載されるかを選ぶこともない。クリエイティブの URL とクリアリング価格を受け取り、それを配信するだけである。

このアーキテクチャは広告主のリーチに最適化されており、パブリッシャーの価値には最適化されていない。パブリッシャーのコンテンツ — ユーザーがそこにいる理由 — は、他者のターゲティングモデルにおけるシグナルの一つに矮小化されてしまう。京都の旅行について丁寧にリサーチされた記事も、セレブのゴシップについてのクリックベイト記事も、exchange にとっては単に Cookie を持つユーザーを運ぶ異なる URL に過ぎない。

### 広告は人々が嫌うものになった

旅行雑誌の広告を嫌う人はいない。しかし人々はアドブロッカーをインストールし、広告を避けるためにプレミアムサブスクリプションに課金し、ウェブ広告をインターネットで最悪のものだと形容する。何が変わったのか？

違いは広告の存在そのものではない — 広告が何になったかである。トレイルガイドの隣にあるハイキングブーツの雑誌広告は自然に感じられる。昨日見た靴のウェブ広告が、無関係なニュース記事にまで追いかけてくるのは侵入的に感じられる。前者はコンテキストに沿ったレコメンデーションであり、後者は監視である。

従来のアドテクは、コンテンツではなく**ユーザー**をターゲットにする。昨日カーディーラーのウェブサイトを訪れたユーザーは、今日訪れるあらゆるサイト — 料理ブログ、ニュースサイト、フォーラム — で車の広告にリターゲティングされる。パブリッシャーのコンテンツは無関係だ。広告がユーザーを追いかけているのだ。その結果、関係者全員にとって違和感のある体験が生まれる：読者はストーキングされているように感じ、パブリッシャーは自分のページに何が表示されるかのコントロールを失い、広告主は間違ったコンテキストで人々を不快にさせるために費用を払う。

これは同意バナーや Cookie ポリシーで解決すべきプライバシーの問題ではない。設計の問題である。ウェブ広告はコンテンツではなく人をターゲットにすることを選び、そうすることで、読者が許容できる — 時には感謝すらできる — ものだった広告を、読者が積極的に抵抗するものに変えてしまったのだ。

### 中小パブリッシャーは取り残されている

プログラマティックスタックには膨大な固定コストがかかる。SSP との統合には技術的な専門知識が必要である。exchange の品質閾値を満たすには最低限のトラフィック量が求められる。レベニューシェアは積み重なる：exchange が一部を取り、SSP が一部を取り、DSP が一部を取り、DMP が一部を取る。各仲介者がトランザクションから上前をはね、パブリッシャーは残りを受け取る。

ニッチなパブリッシャー — 日本の旅行ブログ、専門的な料理サイト、地方のニュースメディア — にとって、この経済性は成り立たない。トラフィックが少なすぎて exchange が相手にしない。コンテンツが特化しすぎていて、広範な行動ターゲティングに向かない。読者のプライバシー意識が高すぎて、侵入的なトラッキングが通用しない。こうしたパブリッシャーは、低品質な広告ネットワークからのわずかな収入を受け入れるか、サイトを低品質な広告で埋め尽くすか、あるいはマネタイズ自体を諦めるしかない。

### レイテンシという税金

リアルタイムビッディングは、すべてのページロードにレイテンシの下限を課す。ブラウザは広告リクエスト、exchange とのラウンドトリップ、クリエイティブのダウンロードを待ってから広告枠をレンダリングしなければならない。高速なサイトでは、広告がページ上で最も遅い要素になる。Header Bidding — パブリッシャーの収益改善を目的とした業界の試み — は、複数のオークションを順次または並列に実行し、それぞれがネットワークのラウンドトリップを追加するため、事態をさらに悪化させた。

ユーザーはそれに気づいている。アドブロッカーの普及率は、アドテクによるページロードの劣化と直接的に相関している。ユーザー体験を重視するパブリッシャーは、そのユーザー体験を収益化するはずのシステムに参加することでペナルティを受けているのだ。

## チャンス：雑誌が正しかったこと

よく作られた雑誌 — 旅行雑誌、料理の出版物、建築ジャーナル — を手に取ってみてほしい。広告を見てみてほしい。関連性がある。旅行雑誌には航空会社、スーツケース、ホテルの広告が載っている。料理雑誌にはキッチン用品、特産食材、料理学校の広告が載っている。広告がそこにあるべきものだと感じられる。コンテンツを邪魔するのではなく、補完している。

そして人々は実際にそれらを見ていた。広告が乱雑でなく、ポップアップや自動再生動画で注意を奪い合っておらず、読んでいる内容に合った、適度に配置されたわずかな広告しかないとき — 人はそれに目を留めた。時にはページを切り取って保管することさえあった。広告に価値があったのは、それが関連性があり、節度があり、読者の注意を尊重していたからだ。

誰もこれらの広告を不気味だとは思わなかった。雑誌がなぜ自分が旅行に興味があると知っているのか不思議に思う人はいなかった。答えは明白だった：旅行雑誌を買ったからだ。コンテンツがシグナルだったのだ。

このモデルは1世紀以上にわたって機能してきた。広告主は、読者が自社のオーディエンスに合致するメディアへの掲載に対してプレミアムを支払った。パブリッシャーはどの広告が掲載されるかをキュレーションし、品質と関連性を維持した。読者は広告を体験の一部として受け入れ — 時には価値あるものとして — 評価した。なぜなら、広告が実際に関心のあるものについてのもので、編集のコンテキストを尊重する形で提示されていたからだ。

そしてインターネットがそれをすべて捨て去った。

広告をコンテンツにマッチさせる代わりに、ウェブ広告業界は広告をユーザーにマッチさせることを選んだ。「この人は何を読んでいるのか？」と問う代わりに、「この人は誰で、何をしてきたのか？」と問うようになった。ページのコンテンツは無関係になった。読者はターゲットになった。そしてパブリッシャー、読者、さらにはほとんどの広告主にとっても、体験全体が悪化した。

**Promovolve は、雑誌が持っていたものを取り戻す試みである。** 関連性のある広告を、コンテンツにマッチさせ、トラッキングも監視も読書体験の劣化もなく提供する。

### なぜ今なのか？

コンテンツベースの広告は新しいアイデアではないが、それをウェブスケールで自動的に行うことは最近まで非現実的だった。ページコンテンツを広告カテゴリに分類するには、手動のキュレーション（高コスト）か原始的なキーワードマッチング（不正確）が必要だった。

LLM がこれを変えた。Gemini Flash への1回の API 呼び出しは1セントの何分の一かのコストで、IAB Content Taxonomy カテゴリを信頼度スコア付きで返す — 日本アルプスのハイキングに関するページを、Travel、Outdoor Recreation、Asia destinations に正確に分類する。これは5年前には経済的に実現不可能だった。今は可能だ。

## Promovolve が違うこと

Promovolve は、パブリッシャーの視点から広告配信を再構築し、コンテンツを唯一のターゲティングシグナルとして使用する。

### オークションはユーザーが来る前に行われる

従来のシステムは、リアルタイムでユーザーを評価する必要があるため、ページロードのたびにオークションを実行する。Promovolve はその必要がない — コンテンツはページロード間で変わらない。そのためオークションはコンテンツが公開または更新されたとき（スケジュールされたクロール経由）に実行され、結果はレプリケートされたインメモリインデックスにキャッシュされる。ユーザーが来たときには、広告はすでに選ばれている。配信レイテンシは50〜200msから1ms未満に低下する。

### 単一の勝者ではなく、複数の候補

オークションごとに1つの勝者を選ぶ代わりに、Promovolve は広告枠ごとに複数の候補をショートリストし、すべてをキャッシュする。配信時に Thompson Sampling がそれらの中から選択し、探索（新しいクリエイティブを試してクリック率を学習する）と活用（最もパフォーマンスの高いクリエイティブを配信する）のバランスを取る。システムはどの広告がどのコンテンツで最も効果的かを継続的に学習し、A/B テストインフラや手動の最適化は不要である。

### 各キャンペーンが入札を学習する

キャンペーンごとの強化学習エージェント（Double DQN）が、15分ごとに自身のパフォーマンス指標 — クリック率、勝率、予算消化率、ペーシング — を観察し、入札倍率を調整する。数日かけて、各キャンペーンは独自の入札戦略を学習する：コンバージョンの高いアグレッシブなキャンペーンはより高く入札し、予算を消化しすぎているキャンペーンは抑制することを学ぶ。手動の入札管理は不要である。

### 予算ペーシングが現実に適応する

自己チューニングゲイン、トラフィック形状学習、振動検出を備えた PI コントローラーが、1日を通じて予算配分を平滑化する。トラフィックが午前10時にピークを迎え午後3時に落ち込むこと、週末は平日と異なるパターンであることを学習し、自動的に調整する。パブリッシャーは、予算が正午までに枯渇して午後中ずっと空枠が残るのではなく、安定した広告配信を目にすることになる。

### ユーザートラッキングなし — なぜなら不要だから

Promovolve はユーザープロファイルを保存せず、トラッキング Cookie を設定せず、クロスサイト識別子を収集しない。ターゲティングは閲覧されているページのコンテンツのみに基づく。これはプライバシーコンプライアンスのための犠牲ではない — 設計の帰結である。広告をコンテンツにマッチさせるとき、ユーザーについて知る必要があることは何もない。コンテンツがすべてを教えてくれる。

## Promovolve は誰のためのものか

### パブリッシャー

Promovolve は以下のようなパブリッシャーのためのものである：

- **読者との関係を自ら管理し**、侵入的なトラッキングでそれを損なわない
- **特定のバーティカルで質の高いコンテンツを制作し**、コンテンツベースのターゲティングが自然に強みを発揮する
- **サブミリ秒の広告配信を求め**、サイトのパフォーマンスを劣化させない
- **シンプルさを好み**、SSP/exchange 統合の運用上の複雑さを避けたい
- **公正な経済性を必要とし**、プログラマティックサプライチェーンの仲介者税なしで運営したい

### 広告主 — 地元企業からグローバルブランドまで

従来のプログラマティックスタックには最低限の規模要件がある。DSP キャンペーンの設定、入札戦略の管理、exchange の最低要件を満たすには、ほとんどの企業を排除するほどの予算と専門知識が必要である。世界のビジネスの大多数 — 地元のレストラン、街の書店、地域のツアーオペレーター、コミュニティイベントの主催者 — は参加できない。

Promovolve は参入障壁をゼロにする。広告主とは、画像とランディングページを持つ誰でものことである。例えば：

- 地域のアウトドアレクリエーションブログに広告を掲載する**地元のハイキング用品店**
- 地域のニュースサイトで日程を告知する**コミュニティフェスティバル**
- 自分の街のフードブログでプロモーションする**料理教室**
- 自分の街について書かれた旅行記事の読者にリーチする**京都の小さなホテル**
- ニッチなパブリッシャーのネットワーク全体でキャンペーンを展開する**グローバルブランド**

統合すべき DSP はない。手動で設定すべき入札戦略もない — RL エージェントが適正価格を学習する。購入すべきユーザープロファイルもない。ただ「これが私の広告で、これが予算で、これが私の製品です」と伝えるだけ。広告主はad productカテゴリ（例: 「Travel」や「Kitchen Equipment」）を選び、システムが公式IABのAd Product Taxonomy 2.0とContent Taxonomy 2.1のマッピングを使って、マッチするcontent categoryを自動的に導出する。あとはシステムが処理する。

これは雑誌広告の仕組みそのものだ。地元のレストランが街の雑誌に4分の1ページの広告を出せた。規模がビジネスに合っていた。Promovolve はそのアクセシビリティをウェブにもたらす。

### 広告代理店

広告代理店はパブリッシャーのプラットフォームを所有していない — クライアントに代わってキャンペーンを管理し、パブリッシャーのサイトに広告を掲載する。従来のプログラマティックの世界では、代理店は DSP 契約、exchange のシート ID、プラットフォーム固有の入札管理ツールの迷路を渡り歩かなければならず、それぞれが途中でマージンを取っていく。

Promovolve を使えば、代理店は広告配信インフラそのものを自ら所有できる。代理店は自社の Promovolve インスタンスを運用し、パブリッシャーとのネットワークを構築し、すべてのクライアントのキャンペーンをエンドツーエンドで管理するシステムをコントロールできる。DSP の仲介者なし。exchange の手数料なし。他者のプラットフォームへの依存なし。代理店がプラットフォームになるのだ。

RL エージェントがキャンペーンごとの入札最適化を処理するため、代理店は手動の入札調整ではなく、戦略とクリエイティブに時間を使える。これは代理店が育ってきた雑誌広告の営業モデルに近い — メディアを選び、掲載枠を選び、クライアントの予算を管理する — ただし今では、それをすべて機能させる技術を代理店自身が所有しているのだ。

## 本書の構成

本書の残りの部分では、ソースコードから導き出されたすべてのアルゴリズムの詳細を文書化する：

- **Architecture** — Pekko クラスタートポロジー、エンティティ階層、データフロー
- **Auction** — 定期バッチオークションの5つのフェーズ
- **Serving** — Thompson Sampling、コールドスタート、配信時のフェア選択
- **Pacing** — PI 制御、自己チューニング、トラフィック形状学習
- **RL** — DQN エージェントの状態空間、報酬関数、トレーニングループ
- **Distributed State** — ServeIndex のレプリケーションと一貫性
- **Comparison** — 従来の SSP/DSP/exchange パターンとのポイントごとの比較

各章は独立して読める。すべての数式、閾値、定数は Scala のソースコードに直接由来する。
