# Bucketed LWWMap設計

ServeIndexはCRDTのデルタサイズを小さく保つために、エントリを**32個のバケット**に分割します。

## なぜバケットなのか？

すべてのエントリを含む単一の`LWWMap`は、任意の変更時に大きなデルタを生成します。バケット分割によりキースペースを分割します：

```
bucket = abs(key.hashCode) % 32
```

各バケットは独立した`LWWMap`です。バケット7への更新は、バケット7のデルタのみを生成します。

## LWWMap (Last-Writer-Wins Map)

コンフリクトはタイムスタンプで解決されます — より新しいタイムスタンプを持つ値が勝ちます。これが安全な理由は：
1. オークション結果はオークション自体によってタイムスタンプが付けられる
2. 新しいオークションは常に古いオークションを上書きすべきである
3. 同じスロットに対する並行オークションは不可能である（AuctioneerEntityはsiteIdでシャーディングされている）

## バケット数：なぜ32なのか？

- **少なすぎる場合**（4）：バケットあたり最大25%のエントリ → 大きなデルタ
- **多すぎる場合**（1024）：CRDTの管理オーバーヘッドが節約分を上回る
- **32**：10,000エントリの場合、バケットあたり約312。バランスが取れている。

## パブリッシャーごとのネームスペース

複合キー`"siteId|slotId"`はエントリを自然にパブリッシャーごとに分割します。異なるサイトのスロットはハッシュの分布により（通常は）異なるバケットに配置されます。

## DData gossipへの影響

32個のバケットとmax-delta-elements 500の場合：
- 各gossipラウンドで、すべてのバケットにわたり最大500の変更を伝播可能
- 10スロットを更新する単一のオークションは、最大10個のバケットに影響
- 他のバケットのgossipは影響を受けない
