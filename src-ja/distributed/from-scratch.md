# 分散ステートをゼロから構築する

ユーザーがページを読み込むとき、Promovolveは1ミリ秒以内に適切な広告を見つける必要があります。なぜデータベースに問い合わせるだけではダメなのでしょうか？

この章では、Promovolveがデータベースの代わりにレプリケートされたインメモリステートを使う理由、リーダーなしで複数のコピーを同期する仕組み、そして「結果整合性」が単に許容範囲というだけでなく、広告配信にとって実際に正しい選択である理由を説明します。

## レイテンシの問題

PostgreSQLのクエリは、高速なローカル接続で1〜5ミリ秒かかります。速そうに聞こえます。しかし、広告配信パスはすべてのページ読み込みで、すべての広告枠で、すべてのユーザーに対して実行されます。毎秒1,000リクエストで各3広告枠の場合、毎秒3,000回のデータベースクエリになり、それぞれがユーザーのページ読み込みにレイテンシを追加します。

さらに悪いことに、データベースのレイテンシにはロングテールがあります。中央値は2msかもしれませんが、99パーセンタイルは20msになることもあります。1つの遅いクエリがレスポンスをブロックします。負荷がかかると、コネクションプールの競合がさらにレイテンシを増加させます。ページパフォーマンスを重視するパブリッシャーにとって、広告配信レイテンシの1ミリ秒ごとが読者への負担になります。

Promovolveの目標：**1ミリ秒以内に広告を配信する**。負荷がかかった状態でこれを安定的に実現できるデータベースはありません。データはリクエストを処理するマシンと同じマシンのメモリ上に存在する必要があります。

## 明白な解決策（とその問題点）

「ローカルキャッシュに入れればいい。」オークション結果を各APIノードの`HashMap`に読み込む。読み取りはナノ秒。問題解決。

そう簡単ではありません。Promovolveは複数ノードのクラスタとして動作します（信頼性とスループットのため）。ノードAがオークションを実行してローカルキャッシュを更新しても、ノードBとCはそれを知りません。リクエストがノードBに到達したユーザーは古いデータ、あるいはデータがまったくない状態になります。

すべてのノードにデータをレプリケートする必要があります。問題は：どうやって？

## 選択肢1：リーダーベースのレプリケーション

従来のアプローチ：1つのノードが「リーダー」（または「プライマリ」）です。すべての書き込みはリーダーを通じて行われます。リーダーがフォロワーにレプリケートします。

```
Write → Leader → Follower 1
                → Follower 2
                → Follower 3
```

これはPostgreSQLのレプリケーション、Redis Sentinel、およびほとんどのデータベースの動作方式です。**強い一貫性**を提供します — 各書き込み後、すべてのノードが同じデータを見ます。

問題点：
- **リーダーがボトルネックになる。** すべての書き込みが1つのノードを通過します。そのノードが遅い、またはダウンすると、書き込みが停止します。
- **リーダー障害時に選出が必要。** 死んだリーダーの検出、新しいリーダーの選出、追いつきに数秒かかります。その間、書き込みは失敗します。
- **ネットワーク分断が厄介。** リーダーが一部のフォロワーに到達できない場合、書き込みを受け付け続ける（乖離のリスク）か、書き込みを停止する（可用性の犠牲）かを選ばなければなりません。

課金システムを支えるデータベースなら、これらのトレードオフは価値があります。5分ごとにリフレッシュされる広告配信キャッシュには、過剰です。

## 選択肢2：リーダーなしでレプリケートする

すべてのノードが書き込み可能で、書き込みが自動的にすべての他のノードに伝播されたらどうでしょうか？

```
Node A writes → gossip → Node B receives
Node B writes → gossip → Node A receives
```

リーダーなし。選出なし。単一障害点なし。すべてのノードがローカルに書き込みを受け付け（高速）、バックグラウンドで他のノードと同期します（結果的に）。

問題：2つのノードが同時に同じキーに異なる値を書き込んだらどうなるか？ リーダーがあれば、これは起こりません — すべての書き込みが1つの場所を通過するからです。リーダーがなければ、コンフリクトを自動的に解決する方法が必要です。

## CRDT：自己マージするデータ構造

**CRDT**（Conflict-free Replicated Data Type）は、並行する書き込みが常にコンフリクトなしにマージできるように設計されたデータ構造です。マージは決定的です — 更新がどの順序で到着しても、すべてのノードが同じ結果に収束します。

最も単純な例：**カウンター**。「count = 5」と格納する代わりに、各ノードが「自分の貢献分はX」と格納します：

```
Node A: my_count = 3
Node B: my_count = 2
Merge: total = 3 + 2 = 5
```

両方のノードが自分のカウンターをインクリメントします。マージは単に合計するだけです。コンフリクトは不可能です。

Promovolveはより洗練されたCRDTを使用します：**LWWMap**（Last-Writer-Wins Map）。各エントリにタイムスタンプを持つキーバリューマップです。2つのノードが同じキーに異なる値を書き込んだ場合、より新しいタイムスタンプを持つ方が勝ちます：

```
Node A writes: key="ad-slot-1" → creative_X at t=1000
Node B writes: key="ad-slot-1" → creative_Y at t=1003
Merge: creative_Y wins (later timestamp)
```

これはシンプルで予測可能です。「コンフリクト解決」は単に「最も新しい書き込みが勝つ」— 変数を上書きするのと同じセマンティクスです。

## PromovolveでのDDataの使い方

Pekkoの**Distributed Data (DData)**はCRDTをgossipプロトコルで実装しています。数秒ごとに、各ノードがデータの変更をランダムなピアと共有します。変更はソーシャルネットワークでの噂話のようにクラスタ全体に伝播し、最終的にすべてのノードに到達します。

Promovolveの**ServeIndex**は、配信パスが必要とするオークション結果を格納します：

```
Key: "site-123|slot-banner-top|bucket-7"
Value: LWWMap of creative candidates
  "creative-abc" → ServeView(assetUrl, cpm, ctr_stats, expires_at, ...)
  "creative-def" → ServeView(...)
```

`AuctioneerEntity`がオークションを完了すると、`WriteLocal`でDDataに結果を書き込みます — 書き込みはローカルノードで即座に完了します。2秒以内（gossip間隔）に、他のノードが更新を受け取ります。

配信パスが広告を選択する必要があるとき、ローカルのDDataレプリカから読み取ります — ローカルのインメモリルックアップで、ネットワークホップはありません。

## なぜ「結果整合性」がここでは問題ないのか

「結果整合性」は怖く聞こえます。ユーザーが古いデータを取得したらどうするのでしょうか？

広告配信において、実際に「古い」とは何を意味するか考えてみましょう：

**クリエイティブが2秒前に更新されたが、このノードはまだgossipを受け取っていない。** ユーザーは以前のクリエイティブを見ます。これは問題でしょうか？ いいえ — そのクリエイティブは2秒前には有効でした。有効なトラッキングURLを持つ正当な広告です。ユーザーが違いに気づくことはありません。

**キャンペーンの予算が尽きたが、配信ノードはまだそのクリエイティブを表示している。** ペーシングゲートと予算予約がこれをキャッチします。ServeIndexに古いエントリがあっても、予算チェック（CampaignEntityに直接問い合わせる）が配信を拒否し、次の候補にフォールバックします。古いキャッシュエントリは無害です。

**オークションが実行されて新しい候補が生成されたが、このノードにはまだ古い候補がある。** 古い候補は最大5分遅れです（再オークション間隔）。前回のオークションの有効な勝者でした。gossipが到着するまでさらに2秒間それらを配信することは問題ありません。

重要な洞察：**配信パスは設計上、近似的です**。Thompson Samplingがランダム性を追加します。ペーシングは確率的にスロットルします。クリックスルー率は推定値です。すでに統計的推定で動作しているシステムに2秒のgossip遅延を追加しても、結果を意味のある形で劣化させません。

強い一貫性は、あなたが必要としない保証を、あなたが実際に必要とするもの — 速度 — を直接損なうコスト（リーダーのボトルネック、ノード間の調整レイテンシ）で提供します。

## バケット分割：CRDTを小さく保つ

LWWMapの1つの問題：数千のエントリを1つのマップに入れると、すべてのgossipサイクルでデルタ全体（前回の同期以降のすべての変更）が送信されます。多くのクリエイティブにわたる頻繁な更新があると、デルタが大きくなります。

Promovolveは各ネームスペースをキーのハッシュによって**32個のバケット**に分割します：

```
bucket = hash(creativeId) % 32
key = "site-123|slot-banner|bucket-7"
```

各バケットは独立したLWWMapです。10個のクリエイティブを更新するオークションは、32個すべてではなく、おそらく8〜10個のバケットに触れます。gossipは変更のあったバケットのみを送信します。これによりデルタサイズが小さく保たれ、gossipが効率的になります。

なぜ32？ バランスです。バケットが多いほどデルタは小さくなりますが、管理するDDataキーが増えます。少ないほどデルタは大きくなりますが、管理はシンプルになります。32は、サイトあたり数十から数百のクリエイティブという典型的なケースに適しています。

## 書き込み一貫性：高速な書き込み、安全な削除

すべての書き込みが同じではありません。Promovolveは操作に応じて異なる一貫性レベルを使用します：

**書き込み（Put、Append、CPM更新）：`WriteLocal`**

書き込みはローカルノードで即座に成功します。gossipがそれを伝播します。gossipの前にローカルノードがクラッシュすると、書き込みは失われますが、次のオークション（5分以内）がそれを再投入します。

これはホットパスに対する正しいトレードオフです。オークション結果は一時的で頻繁にリフレッシュされます。耐久性よりも速度が重要です。

**削除（キャンペーン削除、クリエイティブ削除）：`WriteMajority`**

クリエイティブの削除はすべてのノードで速やかに反映されるべきです — 一時停止されたキャンペーンの広告が、1つのノードが削除を見逃したために配信され続けるのは望ましくありません。`WriteMajority`は確認を返す前に、過半数のノード（例：3つ中2つ）からの確認応答を待ちます。

`WriteMajority`がタイムアウト（800ms）した場合、Promovolveは200msのバックオフで最大5回リトライします。配信すべきでない広告の削除は、速度よりも重要です。

## ノード再起動時はどうなるか？

DDataはインメモリです。ノードが再起動すると、ローカルレプリカは空です。何が起こるでしょうか？

**マルチノードクラスタの場合**：再起動したノードはDDataのanti-entropyプロトコルを通じて他のノードから完全な状態を受け取ります。1回のgossipサイクル（2秒）以内に、完全なレプリカを持ちます。

**シングルノードクラスタの場合（開発環境）**：データは失われます。次の`PeriodicReauction`タイマー（5分以内）がオークションを再実行し、ServeIndexを再投入します。その間、配信パスは`NoContent`（HTTP 204）を返します — 広告枠は空です。理想的ではありませんが、限定的です。

Promovolveは意図的にServeIndexをディスクに永続化**しません**（LMDBを使用する`shard-*`キーとは異なります）。ホットな配信パスを永続化すると、すべてのオークション書き込みにディスクI/Oが追加され、サブミリ秒読み取りのためのインメモリステートの目的が損なわれます。5分間のリカバリウィンドウは許容可能なトレードオフです。

## 全体像

```
Auction completes
  → AuctioneerEntity writes to ServeIndex (WriteLocal, ~0ms)
  → Gossip propagates to all nodes (~2 seconds)
  → Every node has candidates in local memory

User loads page
  → API node reads local ServeIndex replica (~0.1ms)
  → Pacing gate + Thompson Sampling (~0.1ms)
  → Ad response sent (<1ms total)
```

配信パスにデータベースなし。読み取りにネットワークホップなし。ボトルネックになるリーダーなし。遅延を引き起こす選出なし。システムは強い一貫性（不要なもの）を速度（必要なもの）と引き換えにしています。
