# 学習メカニズム

従来のアドテクとPromovolveの両方がフィードバックから学習しますが、レベルとタイムスケールが異なります。

## 従来：RTBフィードバックループ

### DSP側
- **入札値**：勝敗通知からインプレッションの価値を学習
- **オーディエンスターゲティング**：コンバージョンからどのユーザーセグメントが価値があるかを学習
- 学習はクリエイティブレベルではなく**入札レベル**で行われる

### Exchange側
- **Floor price**：入札分布からの学習により、最低受入入札額を設定

### 制限事項
- 探索なし：勝利からのみ学習し、代替クリエイティブがより良いパフォーマンスを発揮するかは決して発見しない
- ユーザー依存：cookie、プロフィール、クロスサイトトラッキングが必要

## Promovolve：3層の学習

### レイヤー1：Thompson Sampling（リクエストごと）

**対象**：特定のスロットに対してどのクリエイティブが最もパフォーマンスが良いか
**タイムスケール**：すべてのインプレッションが`CreativeStats`を更新（1分バケット、60分ウィンドウ）
**方法**：ベイズ事後更新 `Beta(clicks+1, impressions-clicks+1)`

```
Before impression: Beta(5, 95)  → mean 5%
User clicks:       Beta(6, 95)  → mean 5.9%
```

これは最速のループです — サブ秒のフィードバックが次の配信判断に組み込まれます。

### レイヤー2：DQN入札最適化（15分ごと）

**対象**：予算、ペーシング、パフォーマンスを考慮した最適な入札乗数
**タイムスケール**：15分の観察ウィンドウ、約8日で収束
**方法**：Double DQN、8次元ステート、5アクション、クリック報酬 - 過剰支出ペナルティ

```
State → Q-network → action → bidMultiplier adjustment
```

中間タイムスケールで、競争ダイナミクスとペーシングパターンを学習します。

### レイヤー3：カテゴリランキング（オークションごと）

**対象**：各サイトでどのコンテンツカテゴリが価値があるか
**タイムスケール**：オークションごとに更新、7日の半減期減衰、14日の最大経過時間
**方法**：Thompson Sampling、`Beta(prior_α + clicks, prior_β + non_clicks)`

```
Category "sports" on site X:
  Prior: Beta(1, 1) → weight sampled ~0.5
  After data: Beta(15, 95) → weight sampled ~0.14
```

最も遅いループで、サイトレベルのカテゴリ親和性を学習します。

### レイヤー4：トラフィックシェイプ（日次）

**対象**：時間帯別のトラフィックパターン（平日/週末を分離）
**タイムスケール**：日次ロールオーバー、`dayAlpha = 0.2`のブレンド
**方法**：24バケットEMA、`alpha = 0.1`

ペーシングターゲットを実際のトラフィック分布に合わせて調整します。

### レイヤー5：PI自己チューニング（連続）

**対象**：過剰ペース修正の最適な積極性
**タイムスケール**：20サンプルごと、最小500ms間隔
**方法**：持続的な過剰支出検出に基づいてoverpace multiplierが1.5xから5.0xの間で適応

## 比較

| 次元 | 従来のRTB | Promovolve |
|-----------|----------------|------------|
| 最適化対象 | 入札価格 | クリエイティブ + 入札 + カテゴリ + ペーシング |
| 探索 | なし | 組み込み（Thompson Sampling） |
| 学習レイヤー | 1（入札レベル） | 5（リクエストごとから日次まで） |
| ユーザーターゲティング | あり（プロフィール） | なし（コンテンツベース） |
| プライバシー影響 | 高い（トラッキング） | 低い（ユーザープロフィールなし） |
| コールドスタート | 過去の入札データ | ベイズ事前分布 + ラウンドロビンウォームアップ |
| 適応性 | リアルタイム入札 | RLが日数で収束、TSがリクエストごとに適応 |
